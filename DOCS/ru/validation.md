# Валидация — тестирование, аудит, эффективность, ресурсы

Всё здесь замерено на реальном оборудовании — роутер Keenetic ARMv8 (ARM64) quad-core,
1 GB RAM, принимающий реальный трафик с отдельной cloud-VM на публичный WAN роутера. Оба
конца принадлежат оператору; атакующий хост на провайдере, разрешающем security-тестирование.
**Все адреса замаскированы** документационными диапазонами RFC 5737 (WAN роутера
`198.51.100.10`, attacker `203.0.113.45`); цифры только агрегированные.

## 1. Регрессионный набор

`tacet-test.sh` — самодостаточный end-to-end набор (58 проверок), выполняемый на
работающем роутере. Безопасен на боевом роутере: каждая мутация использует
документационные диапазоны, config снапшотится и восстанавливается, все тестовые записи
чистятся.

Покрывает: read-эндпоинты отдают валидный JSON; валидацию адресов (octets 0–255, mask
≤ 32); роутинг CIDR-бана (CIDR с closed/open-поверхности попадает в subnet-сет) и его
жизненный цикл; whitelist-notes и их очистку; устойчивость import
(враждебный ввод, control-байты, GET-vs-POST, пустое тело, override-пустой-секции);
backup round-trip; валидацию и клампинг config; конкурентность; регрессию info-badge; и —
через SSH — целостность netfilter-правил, длину hashlimit-имён, корректность сортировки
баз, чистый выход collector, guard reboot-persistence и сходимость счётчиков engine/CGI.
Результат: **58/58 green**, идемпотентно при повторных прогонах. Деструктивные тесты
(wipe-round-trip whitelist, сохранение конфига) выполняются только если их снапшоты
верифицируемо распарсились — сбой экспорта их пропускает, а не рискует живыми данными.

## 2. Security-аудит — поверхность защиты

Три раунда тестов с живым атакующим прогнали каждый механизм. Сведённые результаты:

| Механизм | Тест | Результат |
|---|---|---|
| **Closed-port auto-ban** | скан закрытых портов сверх порога 8/мин | ✅ бан за секунды, в `tacet-scan` |
| **Точность порога** | 6 SYN (под 8), затем +15 (сверх) | ✅ 6 → нет бана, +15 → бан; rate-метр точный |
| **Forwarded-port auto-ban** | флуд проброшенного порта сверх настроенных 45/мин | ✅ бан в `tacet-flood` |
| **Subnet trap (изолированно)** | per-IP trap off, флуд одного IP, subnet-метр 10/мин | ✅ весь `/24` забанен в `tacet-cnet` — subnet-метр срабатывает независимо |
| **Reputation / first-packet ban** | добавить источник в `tacet-threat`, один SYN | ✅ бан с первого пакета, без порога |
| **Эффективность drop** | коннект к *открытому* проброшенному порту, будучи забаненным | ✅ был доступен → таймаут при бане → снова доступен после unban |
| **Sticky ban** | продолжать обращаться, будучи забаненным | ✅ timeout не убывает — обновляется на каждом дропнутом пакете |
| **Reject policy** | `BAN_REJECT=yes`, коннект при бане | ✅ `ConnectionRefused` за **0.02 с** (TCP reset) вместо многосекундного silent-drop таймаута |
| **Whitelist wins** | whitelist атакующего, затем флуд | ✅ никогда не забанен, остаётся доступен — whitelist проверяется раньше ban-правил |
| **Master switch** | переключить off, затем on | ✅ off → все drop-правила ушли, источник доступен; on → правила назад, источник переблокирован, бан пережил переключение |
| **Persistence (симуляция reboot)** | save / flush / restore сета | ✅ бан сохраняется — баны восстанавливаются после reboot |

### DoS-устойчивость

- **conntrack exhaustion** — TCP SYN flood банится за ~3 с; его дропнутые пакеты дальше
  ничего не добавляют, и conntrack остался ограничен (~800 из 32768-записей таблицы).
  Быстрый бан плюс собственные source-port/concurrency-лимиты источника ограничивают его
  footprint.
- **Source spoofing** — raw SYN flood с подделанным источником (47k pps) не достиг
  роутера: cloud egress anti-spoofing его дропает, как дропал бы у реального атакующего.
- **UDP flood** — Tacet только TCP-SYN, но UDP flood на роутер не создал ни одной
  conntrack-записи и заметной нагрузки (дропнут дёшево базовым firewall).

## 3. Нагрузочное тестирование

Единственный значимый источник нагрузки роутера — его собственный web-UI, а не трафик
атаки.

| Источник нагрузки | Load роутера (1-min) |
|---|---|
| 40 параллельных API-запросов (легитимных) | **~7.3** |
| Флуд ~414 пакет/с (забаненный источник) | **~0.35** (idle) |

**API-пропускная vs concurrency** — shell CGI выходит на плато **~14 req/s**; за ~20 параллельных
запросов latency и load растут (запросы встают в очередь), но ни один не падает. В реальном
использовании UI опрашивает раз на открытый таб за `REFRESH_SEC` — десять открытых
dashboard при дефолтных 60 с — это ~0.17 req/s, на два порядка ниже потолка.

**Юзабилити роутера во время устойчивого флуда** — 40-секундный, 80-thread connection
flood (16 282 попытки, ~414 пакет/с дропнуто): источник забанен за 5 с, **UI остался
отзывчивым (~130–150 мс)**, **load оставался на уровне idle (~0.35)**. После бана
источник не добавляет заметной нагрузки, потому что дроп это O(1) ipset-lookup плюс `DROP`.

**Комбинированный стресс** — флуд атаки + принудительный прогон collector + десять
параллельных API-pollers одновременно: **30/30** запросов вернули валидный JSON, пик load
1.43, conntrack в норме ~868/32768. Ни один запрос не упал и не вернул повреждённый
ответ.

## 4. Эффективность в реальной эксплуатации (~20 ч наблюдения)

Снапшот ban-list после ~20 ч работы на реальном WAN-канале — данные о том, что Tacet
ловит реальных атакующих, а не тест:

- **~713 активных банов** (666 одиночных адресов + 47 whole-subnet), в равновесии (новые
  атакующие приходят примерно так же быстро, как истекают старые баны).
- **~314 000 пакетов дропнуто** от забаненных источников, **~970/мин** (за 5.4 ч с
  момента, когда последний rebuild правил обнулил счётчик).
- **63 % атакующих исходят из cloud/hosting** адресного пространства — топ-владельцы
  Microsoft/Azure, Akamai, Alibaba, UCLOUD, DigitalOcean, Amazon, Google. Арендованные VM
  и сканеры доминируют. Топ-страны отражают географию хостинга
  (US ~50 %, NL, DE, GB, SG, FR, CN, JP, RU).
- **86 %** забаненных одиночных IP есть и в публичных threat-feeds — поведенческие traps
  согласуются с reputation-данными.
- остальные **~14 % пойманы только поведением** — активно сканируют этот роутер, но пока
  ни в одном публичном feed. Это то, что reputation-only блоклист пропустил бы.
- По наблюдаемой активности: сотни port-scanners, десятки brute-forcers, плюс web-, mail-
  и VoIP-abuse. Reputation/first-packet баны и subnet-folds оба фигурируют в event log.

## 5. Footprint ресурсов

Резидентный footprint Tacet пренебрежимо мал на фоне ~556 MB свободных.

**Память**

| Компонент | Footprint |
|---|---|
| Все ipsets (ядро) | **~0.6 MB** (threat 0.49 MB / 16.7k IP доминирует) |
| UI web-сервер (`lighttpd`) RSS | ~4.6 MB |
| Базы на flash (geo+asn+threat+activity+tor `.dat`) | ~30 MB |
| Collector — пик RSS во время geo-tick | **~3.6 MB** |
| CGI-запрос — пик RSS (затем выходит) | ~5 MB |

Collector достигает пика ~3.6 MB, потому что merge-join резолвер потоково читает базы
355k+398k строк указателем, а не загружает их в массивы (что было бы десятки MB).

**CPU**

| Операция | CPU (user) |
|---|---|
| Collector — geo-tick (раз в 5 мин) | ~4.4 с |
| Collector — обычная минута | ~0.4 с |
| CGI-запрос | ~0.1–0.5 с |

В среднем collector — значительно меньше 2 % одного ядра на quad-core ARM. Дроп
забаненного трафика бесплатен (ipset-lookup + `DROP`), так что никакой флуд от одного
источника не поднимает load выше idle.
